### 1-4. 배포 절차 (Kubernetes — SSAFY K3s)

```bash
# 1. 네임스페이스 생성
kubectl apply -f infra/k8s/00_namespace.yaml

# 2. Kubernetes Secret 생성 (환경변수를 K8s Secret으로 주입)
kubectl create secret generic modu-secret \
  --from-env-file=.env \
  -n modu-apps

# 3. 인프라 리소스 적용 (DB, MQ, Cache)
kubectl apply -f infra/k8s/postgres.yaml
kubectl apply -f infra/k8s/redis.yaml
kubectl apply -f infra/k8s/mongodb.yaml
kubectl apply -f infra/k8s/kafka.yaml

# 4. 애플리케이션 배포
kubectl apply -f infra/k8s/backend.yaml
kubectl apply -f infra/k8s/kis-gateway.yaml
kubectl apply -f infra/k8s/frontend.yaml
kubectl apply -f infra/k8s/ai-agent.yaml
kubectl apply -f infra/k8s/analysis-server.yaml

# 5. Ingress 적용
kubectl apply -f infra/k8s/ingress.yaml

# 6. 배포 확인
kubectl get all -n modu-apps
```

#### 배포 시 특이사항

- **SSAFY 호스트 Nginx 수동 설정 필요**: K8s Ingress와 별도로 호스트 서버(`k14b106.p.ssafy.io`)의 `/etc/nginx/sites-enabled/default`에 WebSocket `Upgrade`/`Connection` 헤더 전달 설정이 필요합니다. K8s 매니페스트만으로 WebSocket 연결이 해결되지 않았으며 호스트 Nginx도 함께 수정해야 합니다.
- **KIS WebSocket 단일 인스턴스**: `kis-gateway` Pod는 KIS API 앱키 당 WebSocket 1세션 제약으로 인해 반드시 **1개 인스턴스**만 운영해야 합니다 (`scale` 금지).
- **Frontend 빌드 아규먼트**: K8s 배포 시 프론트엔드 이미지는 `--build-arg NGINX_CONF=nginx.k8s.conf`를 사용하여 Nginx 설정을 K8s용으로 교체합니다.
- **Flyway 마이그레이션**: 백엔드 최초 실행 시 Flyway가 자동으로 DDL을 적용합니다. `out-of-order: true`로 설정되어 타임스탬프 순서가 다소 어긋나도 안전하게 적용됩니다.
- **Kafka KRaft 모드**: Zookeeper 없이 KRaft 모드로 운영합니다. `CLUSTER_ID`는 고정값(`MkU3OEVBNTcwNTJENDM2Qg`)을 사용합니다.

## 3. DB 덤프 파일

- PostgreSQL 및 MongoDB 덤프 파일은 별도로 제공됩니다.  
- 최신 덤프 파일은 팀 공유 저장소(Notion 또는 팀 채널)를 참고하세요.

### PostgreSQL 복원

```bash
# PostgreSQL 컨테이너 실행 후 복원
docker exec -i modu-postgres psql -U postgres -d modu_db < modu_db_dump.sql
```

### MongoDB 복원

```bash
# MongoDB 컨테이너 실행 후 복원
docker exec -i modu-mongo mongorestore \
  --username modu --password <password> --authenticationDatabase admin \
  --db modu_mongo --archive < modu_mongo_dump.archive
```
